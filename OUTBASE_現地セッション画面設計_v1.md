# OUTBASE 現地セッション画面設計 v1

## 0. この設計の前提

RC10〜RC12は採用しない。  
理由は、正本照合表で確認したとおり、現地タブを「モード一覧」画面にしてしまっていたため。

正本側の方向はこれ。

> モードを選ばせるのではなく、現地セッションとして記録し、あとでAIが意味づけ・整理する。

つまり、現地タブの最初に「コタ散歩 / キャンプ場散歩 / 設営 / 撤収 / ドライブ」を大きく選ばせる画面にはしない。  
ただし、それらを消すわけではない。  
セッション内の「候補」「記録タイプ」「保存先」として扱う。

---

## 1. 現地タブの正しい役割

現地タブは、説明を読む場所でも、機能一覧を見る場所でもない。

```text
現地タブ =
キャンプ中・散歩中・移動中に、
あとで忘れることを
最小操作で残す場所
```

現地タブでやることは、次の4つに絞る。

```text
・声で残す
・写真で残す
・場所を残す
・あとで整理できる保存先に送る
```

## 2. 現地タブの最初の画面

### 2-1. 画面ラフ

```text
┌────────────────────────┐
│ 現地セッション                    │
│ SP赤城山 / 2026.06.26 / コタあり   │
│ 天気・時刻・現在地取得中             │
└────────────────────────┘

今はこれっぽい
[ コタ散歩中？ ] [ サイト調査？ ] [ 設営中？ ]
[ ドライブ中？ ] [ 撤収中？ ]

大きい操作
┌──────────┬──────────┐
│ 音声メモ     │ カメラ       │
│ 長押しで残す  │ 写真で残す    │
└──────────┴──────────┘
┌──────────┬──────────┐
│ 現在地       │ Google Map  │
│ 場所を残す    │ 地図を開く    │
└──────────┴──────────┘

今すぐ使う候補
[ うんち ] [ おしっこ ] [ 体調 ] [ 危険 ]
[ サイト候補 ] [ 木陰 ] [ 水場 ]

直近の記録 3件
・10:21 音声 → コタ散歩候補 / 非公開ではない
・10:24 うんち → 非公開コタ健康メモ
・10:31 写真 → サイト調査候補
```

### 2-2. 重要な点

ここでは「モードを決定」しない。  
AIが今の状況から候補を出す。

```text
候補の判断材料

・現在の予定
・現在地
・移動速度
・時刻
・GPS移動
・直前の記録
・写真/音声の内容
・コタ同伴フラグ
```

ユーザーは候補を押してもよいが、押さなくても記録できる。  
これが「最小操作」。

## 3. コタ散歩はどう扱うか

### 3-1. ユーザーがすること

ユーザーは「コタ散歩モード」に入る必要はない。  
散歩中に必要なものだけ押す。

```text
・音声メモ
・カメラ
・現在地
・うんち
・おしっこ
・体調
・危険
・犬友達
```

### 3-2. OUTBASE側の判断

以下が出たら、AIはコタ散歩候補として扱う。

```text
・コタ同伴予定
・徒歩速度
・GPSが連続移動
・うんち/おしっこ/体調が押された
・犬友達が押された
・音声に「コタ」「暑い」「疲れた」「危ない」などがある
```

### 3-3. 保存先

```text
うんち       → 非公開コタ健康メモ
おしっこ     → 非公開コタ健康メモ
体調         → 非公開コタ健康メモ
危険         → 危険ポイント
犬友達       → 犬友達メモ
良い道       → 散歩コース候補
写真         → 現地写真 / コタ散歩候補
音声         → 現地メモ / コタ散歩候補
```

### 3-4. 後で使う場所

```text
・コタの体調傾向
・散歩コース候補
・危険ポイント回避
・次回散歩提案
・レビューには出さない非公開メモ
```

## 4. キャンプ場散歩 / サイト調査はどう扱うか

### 4-1. 名前

画面上の表現は「キャンプ場散歩」より、正本寄りには以下が近い。

```text
サイト調査
キャンプ場メモ
場内チェック
```

ユーザーが歩いているだけではなく、次回のために場所を調べているから。

### 4-2. ユーザーがすること

```text
・音声メモ
・カメラ
・現在地
・サイト候補
・景色
・木陰
・水場
・トイレ
・炊事場
・危険
```

### 4-3. OUTBASE側の判断

以下が出たら、AIはサイト調査候補として扱う。

```text
・キャンプ予定中
・キャンプ場内にいる
・徒歩速度
・現在地記録が複数ある
・サイト候補/水場/木陰/危険が押された
・写真が多い
・音声に「次ここ良い」「水場近い」「木陰」「コタに良い」などがある
```

### 4-4. 保存先

```text
サイト候補   → 場所カード / 次回サイト候補
景色         → 場所カード / レビュー候補
木陰         → 場所カード / コタ向け候補
水場         → 場所カード / 設営動線
トイレ       → 場所カード / 設備メモ
炊事場       → 場所カード / 設備メモ
危険         → 危険ポイント
写真         → 場所カード候補
音声         → キャンプ場メモ
```

### 4-5. 後で使う場所

```text
・キャンプ場レビュー
・次回サイト選び
・コタ連れ向け判断
・水場/トイレ距離
・危険ポイント
・再訪時のメモ
```

## 5. 設営はどう扱うか

### 5-1. ユーザーがすること

設営は「設営モードを開いて読む」ではなく、設営中に必要な記録をすぐ残す。

```text
・設営開始
・配置写真
・音声メモ
・現在地
・足りないギア
・使わなかったギア
・困ったこと
・次回改善
・設営完了
```

### 5-2. OUTBASE側の判断

以下が出たら、AIは設営工程候補として扱う。

```text
・チェックイン後の時間帯
・キャンプ場内
・写真が設営/配置っぽい
・音声に「張る」「設営」「配置」「ペグ」「足りない」などがある
・設営開始/完了が押された
```

### 5-3. 保存先

```text
設営開始       → 設営ログ
設営完了       → 設営ログ
配置写真       → 次回レイアウト
足りないギア   → ギア / 買い物 / 準備
使わなかったギア → ギア見直し
困ったこと     → 改善メモ
次回改善       → 改善メモ
```

### 5-4. 後で使う場所

```text
・次回の設営順
・レイアウト再現
・ギア準備
・買い物リスト
・改善リスト
・設営時間の見積もり
```

## 6. 撤収はどう扱うか

### 6-1. ユーザーがすること

```text
・撤収開始
・カメラ
・音声メモ
・乾燥
・忘れ物
・ゴミ
・積込
・壊れた/汚れた
・撤収完了
```

### 6-2. OUTBASE側の判断

以下が出たら、AIは撤収工程候補として扱う。

```text
・チェックアウト前
・撤収開始/完了が押された
・音声に「濡れた」「乾燥」「片付け」「忘れた」「積む」などがある
・壊れた/汚れたが押された
```

### 6-3. 保存先

```text
撤収開始       → 撤収ログ
撤収完了       → 撤収ログ
乾燥           → 帰宅後やること / ギアケア
忘れ物         → 忘れ物チェック
ゴミ           → 撤収ログ
積込           → 車載改善
壊れた/汚れた   → ギアケア / 修理 / 洗浄
写真           → 撤収写真
音声           → 撤収メモ
```

### 6-4. 後で使う場所

```text
・帰宅後やること
・乾燥リスト
・修理リスト
・忘れ物チェック
・車載改善
・次回撤収時間の見積もり
```

## 7. ドライブはどう扱うか

### 7-1. ユーザーがすること

```text
・出発
・休憩
・給油
・買い物寄り道
・渋滞
・道の駅/コンビニ
・到着
・音声メモ
・Google Map
```

### 7-2. OUTBASE側の判断

以下が出たら、AIは移動工程候補として扱う。

```text
・車移動速度
・キャンプ前後の時間帯
・出発/到着が押された
・音声に「渋滞」「コンビニ」「買い物」「道の駅」などがある
```

### 7-3. 保存先

```text
出発           → 移動ログ
休憩           → 休憩候補
給油           → 移動ログ
買い物寄り道   → 買い物/通り道候補
渋滞           → 渋滞メモ
道の駅/コンビニ → 場所カード / 通り道候補
到着           → 移動ログ
```

### 7-4. 後で使う場所

```text
・次回の出発時間
・通り道コンビニ
・道の駅候補
・買い物のタイミング
・渋滞回避
・帰り道判断
```

## 8. 記録後の確認画面

記録直後に、毎回長い画面を出さない。  
ただし、必要なときだけ小さく確認する。

### 8-1. 例：うんちを押した場合

```text
保存しました

うんち
保存先：非公開コタ健康メモ
レビューには出しません

[ OK ]
```

### 8-2. 例：サイト候補を押した場合

```text
場所カード候補に保存しました

サイト候補
保存先：場所カード / 次回サイト候補

[ OK ] [ 名前を付ける ]
```

### 8-3. 例：足りないギアを押した場合

```text
準備に戻します

足りないギア
保存先：ギア / 買い物 / 準備

[ OK ] [ 買い物リストへ ]
```

## 9. 整理画面側の受け皿

現地タブで全部処理しない。  
現地タブは「送る」だけ。

整理画面では以下に分かれる。

```text
・今日の記録
・場所カード候補
・非公開コタ健康メモ
・設営改善
・撤収/帰宅後やること
・ギアケア
・買い物/準備へ戻すもの
・ドライブ/移動ログ
・レビュー候補
```

---

## 10. 実装方針

次に実装するなら、最初の範囲はこれだけ。

```text
実装する：
・現地セッション画面
・AI候補チップ
・音声 / カメラ / 現在地 / Google Map
・文脈別ショートカット
・記録に保存先を付ける
・非公開メモ分離
・直近3件表示

実装しない：
・長い説明
・モード一覧
・線だけ地図
・全履歴表示
・タイマー一覧
・ギア一覧
・レビュー生成
```

---

## 11. 採用条件

この設計を実装する場合、以下を満たさないものは採用しない。

```text
・最初にモード一覧を出さない
・現地セッションとして開始できる
・コタ散歩/サイト調査/設営/撤収/ドライブは候補として扱う
・うんち/おしっこ/体調は非公開保存
・サイト候補/木陰/水場/危険は場所カード候補
・足りないギア/困ったことは準備/ギア/改善へ戻す
・乾燥/壊れた/汚れたは帰宅後/ギアケアへ戻す
・出発/休憩/渋滞/到着は移動ログへ保存
・現地では直近3件だけ表示
・整理は整理画面で行う
```

以上。
