# OUTBASE_RESTORE04.15 実装設計書
## 外出属性UX / デザイン復旧 / Context裏側化 / ルート04.8保護

作成日: 2026-07-10  
扱い: **ZIP作成前の実装設計書。まだZIPは作らない。**  
目的: **04.8の見やすさを正として、04.13の内部モデルと外出属性LOCKを裏側に統合する。04.14のUI劣化は捨てる。**

---

## 0. 最終結論

04.15は「機能を増やす版」ではない。  
04.15は、壊れたUXとデザインを復旧しながら、今後の外出OS化に必要な最小データ構造だけを裏側へ入れる版。

```text
正: 04.8のルート一画面性 / OUTBASE感 / 情報密度
採用: 04.13のContext Model最小土台
採用: 外出タイプ網羅LOCK v1の属性テンプレート方式
破棄: 04.14の大型説明カード / 汎用カードだらけ / ダサいタブホーム
```

04.15で画面に見せるものは少なくする。

```text
OUTBASE
表示中：〇〇　⌄
保存先：〇〇

下バー：予定 / 探す / 準備 / 記録 / 思い出
```

Context Modelや外出属性は、ユーザーに説明カードとして見せない。  
必要な入口を出し分けるための裏側ルールとして使う。

## 3. データ設計

### 3.1 state追加・維持方針

04.15では既存stateに最小追加する。大規模DB刷新はしない。

```js
state.context = {
  viewTargetId,
  viewTargetType,
  recordTargetId,
  recordTargetType,
  primaryRecordingSessionId,
  activeSessionIds,
  referenceTargetIds,
  inboxId
}

state.sessions = []
state.links = []
```

04.13で導入済みの考え方は維持するが、画面に説明として出さない。

### 3.2 Plan属性

既存events/planに対し、必要なものだけ`attributes`として持つ。

```js
plan.attributes = {
  templateType,
  category,
  stayType,
  transport,
  petRelated,
  petAccompanied,
  targetPets,
  companions,
  reservation,
  ticket,
  weatherImpact,
  routeImportance,
  gearImportance,
  foodImportance,
  recordPurpose,
  status
}
```

### 3.3 属性の目的

属性は画面に全部出すためではない。  
以下を判断するために使う。

```text
・準備タブで何を出すか
・記録タブで何を優先するか
・ルートが必要か
・持ち物/買い物/料理/ギア/天気の優先度
・ペット同伴あり/なし/未定で出す入口の違い
```

## 4. 関数設計

### 4.1 追加する関数

```js
ob415NormalizePlanAttributes(plan)
```

役割:
- 既存予定に属性がない場合の補完
- キャンプ/散歩/イベント等の初期推定
- ただし勝手に強制確定しない

```js
ob415PlanAttributes(plan)
```

役割:
- 対象planの属性を安全に返す
- 未定値を含めて扱う

```js
ob415EntrySet(plan, tabName)
```

役割:
- 予定/探す/準備/記録/思い出ごとの入口候補を返す
- 外出タイプ名ではなく属性で判断する

```js
ob415RouteMode(plan)
```

役割:
- ルート画面の表示モードを返す
- camp / drive_walk / event / generic など
- 04.8のルート一画面性を崩さない

```js
ob415MiniStatusHtml()
```

役割:
- 表示中/保存先を小さく表示
- Context説明は出さない

```js
ob415TabHomeHtml(tabName)
```

役割:
- 各タブホームを入口として描画
- 説明カードではなくアクション入口

### 4.2 04.13から使う関数

以下は内部部品として使う。

```text
ob413NormalizeContext
ob413RecordTarget
ob413ActiveSession
ob413StartSession
ob413EndPrimarySession
ob413Link
ob413LinksFor
```

ただし、以下は常時大きく表示しない。

```text
ob413ContextStatusHtml
```

使う場合も、小型ステータスへ吸収する。

## 5. UI設計

## 5.1 共通ヘッダー

### 表示

```text
OUTBASE
表示中：赤城山キャンプ　⌄
保存先：コタ場内散歩
```

### ルール

```text
・1〜2行まで
・大型カードにしない
・Context説明文を出さない
・表示中を変えても保存先は勝手に変えない
・保存先不明時だけ未確認箱
```

---

## 5.2 ホーム

ホームは説明ではなく最短入口。

```text
今日なにする？

[ 記録する ]
話す・撮る・場所・メモ

[ 準備する ]
ルート・持ち物・買い物・天気

[ 予定を見る ]
次の外出・候補・今日
```

合格条件:

```text
・3秒で押す場所が分かる
・記録入口が最も押しやすい
・説明カードが並ばない
・04.14のようにダサくならない
```

---

## 5.3 予定タブ

目的: 外出を選ぶ / 作る / 切り替える。

初画面:

```text
次の外出
今日
候補
追加
切替
```

出してよい情報:

```text
・外出名
・日付
・外出の状態
・簡単な属性チップ 1〜2個まで
```

出さない情報:

```text
・属性一覧
・親子孫ツリー
・Context説明
```

---

## 5.4 探すタブ

目的: 次に行く場所・イベント・買い物候補を探して保存する。

入口:

```text
キャンプ場
散歩場所
イベント
ドッグカフェ
ショップ
下見候補
保存候補
```

補足:

```text
・ペット関連イベントは同伴あり/なし/未定で扱える
・探す時点ではペット同伴が未定でもよい
・候補保存後に予定化できる
```

---

## 5.5 準備タブ

目的: 表示中の外出に必要な準備を出す。

共通入口:

```text
ルート
持ち物
天気
メモ
```

属性別の追加入口:

```text
キャンプ:
  買い物 / 料理 / ギア / 設営 / 撤収

通常散歩:
  散歩開始 / 持ち物 / 近所メモ

ドライブ散歩:
  ルート / 駐車場 / 現地散歩開始 / 水 / 休憩

イベント同伴あり:
  駐車場 / 入場時間 / 水 / 休憩 / カート / 暑さ寒さ

イベント同伴なし:
  駐車場 / 入場時間 / 見たいブース / 買いたい物 / 写真メモ
```

---

## 5.6 記録タブ

目的: 今の行動を保存する。

入口:

```text
話す
撮る
動画
場所
ピン
メモ
散歩開始
```

保存先表示:

```text
保存先：コタ場内散歩
```

ルール:

```text
・保存先は小さく表示
・保存先不明でも記録は止めない
・未確認箱へ仮保存できる
・記録中セッションへ保存する
・表示中プランを見ていても保存先は勝手に変えない
```

---

## 5.7 思い出タブ

目的: 見返す / 整理 / 改善する。

入口:


補足:

```text
・Instagram / note / レビュー投稿整理はここに入る
・帰宅後レビュー、ギア乾燥、片付けもここから繋ぐ
```

### Step 1: ベース作成

```text
04.8 ZIPを作業ベースにする

```

### Step 2: Context Model部品移植

04.13から以下を移植する。

```text
state.context

state.links
ob413NormalizeContext
ob413RecordTarget
ob413ActiveSession
ob413StartSession
ob413Link
```

ただし、Context説明UIは移植しない。

### Step 3: 外出属性モデル追加

```text
ob415NormalizePlanAttributes
ob415PlanAttributes
ob415EntrySet
ob415RouteMode
```

を追加する。

### Step 4: タブホーム復旧

```text
home / prep / discover / memory / field の入口を最小化
説明カードを削除
各タブは「入口」を出す
```

### Step 5: ルート復旧

```text
04.8のroutePrepViewを基準にする
Context表示は小型チップのみ
大型カード禁止
```

### Step 6: 記録保存先統一

```text
写真 / 動画 / 音声 / メモ / GPS / ピンは ob413RecordTarget 経由
保存先不明時は未確認箱
```

### Step 7: 検証

下記シナリオを通してからZIP。

## 9. 検証シナリオ

### 9.1 通常散歩

```text
表示中：コタ通常散歩
保存先：コタ通常散歩セッション
準備：散歩開始 / 持ち物 / 近所メモ
記録：話す / 撮る / ピン / メモ
ルート：必須ではない
```

合格:

```text
キャンプ用チェックイン/買い物/設営が出しゃばらない
```

### 9.2 ドライブ散歩

```text
表示中：手賀沼ドライブ散歩
保存先：現地散歩セッション
準備：ルート / 駐車場 / 水 / 休憩
記録：現地散歩開始 / 写真 / ピン / メモ
```

合格:

```text
ルートは使えるが、キャンプのチェックイン逆算に寄りすぎない
```

### 9.3 キャンプ

```text
表示中：赤城山キャンプ
保存先：赤城山キャンプまたは場内散歩セッション
準備：ルート / 買い物 / 料理 / ギア / 天気
ルート：04.8の一画面性
```

合格:

```text
04.8より見にくくならない
```

### 9.4 ペット関連イベント・同伴あり

```text
表示中：ペットイベント
属性：petRelated true / petAccompanied true
準備：ルート / 駐車場 / 入場時間 / 水 / 休憩 / カート / 暑さ寒さ
記録：写真 / メモ / 買い物 / 会場内メモ
```

合格:

```text
ペット同伴に必要な入口が出る
```

### 9.5 ペット関連イベント・同伴なし

```text
表示中：ペットイベント
```

合格:

```text

```

### 9.6 予定なし記録

